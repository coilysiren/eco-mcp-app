# The actual fix — inline thumbnails as data URIs

Timestamp: 2026-04-20 ~15:52 PT

## What turned out to be the real problem

MCP Apps is NOT broken in Claude Desktop chat UI. Kai's test showed it working
end-to-end: iframe rendered, card grid visible, titles + durations + metadata
displayed correctly. The RPC log shows the full flow completing:

```
23:04:00.644  client → tools/list  id=3
23:04:00.649  server → tools/list result (with _meta.ui.resourceUri)
23:04:00.653  client → resources/read ui://app/preview.html  id=4
23:04:00.661  server → resources/read result (308KB of HTML)
```

Host fetched the UI resource. Iframe rendered. Everything works.

**The single remaining bug:** external thumbnail URLs (`cdn.example.com`,
`storage.googleapis.com/example-bucket/...`) get blocked by the iframe's
hardcoded `img-src 'self' data: blob: https://assets.claude.ai` CSP.

This is the bug tracked in
[anthropics/claude-ai-mcp#40](https://github.com/anthropics/claude-ai-mcp/issues/40):
Claude.ai ignores `_meta.ui.csp.resourceDomains` and enforces a hardcoded CSP.
@antonpk1 (Anthropic) replied 2026-04-09: "Unfortunately we can't allow nested
iframes on claude.ai at the moment (security concerns)." — same hardcoding
extends to `img-src`.

## The workaround

Since `data:` URIs are always permitted by the sandbox CSP, fetch thumbnails
server-side and inline them as `data:image/jpeg;base64,...` before shipping
the payload to the iframe.

### Patch applied to server.ts

Added helpers:
- `inlineThumbnail(url)` — fetches with x-access-token, 2.5s timeout, 500KB cap,
  returns `data:...` or null on failure.
- `inlineThumbnails(projects)` — Promise.all fan-out over all projects.

Call sites:
- `list_items` handler now runs `inlineThumbnails(limited)` before
  building the payload.
- `open_item` handler inlines the single project.

Smoke-test output (all 3 projects tested):
```
'Untitled Project': data URI, 28,591 chars
'Export Webps':     data URI, 34,279 chars
'Untitled Project': data URI, 17,999 chars
```

Total payload overhead: ~200KB for 8 projects. Acceptable — the iframe bundle
itself is 308KB.

### Build

```
/opt/homebrew/bin/npm run build:node
```

No UI rebuild needed — the React code already reads `thumbnailUrl` and passes it
straight into `<img src>`, so data URIs Just Work with the existing bundle.

## To actually see the fix

MCPs only load at Claude Desktop startup. Kai needs to:

1. `Cmd+Q` Claude Desktop (fully quit)
2. Reopen
3. In a fresh chat: `Use sibling-mcp-app to show me my recent items.`
4. Thumbnails should now render inside the card grid.

## Revised conclusions about the whole investigation

- MCP Apps is working in Claude Desktop chat UI today.
- MCP Apps is genuinely missing in Claude Code Desktop (clientInfo
  `local-agent-mode-*` doesn't advertise `io.modelcontextprotocol/ui`). This
  is a separate product gap worth filing as an issue — Kai likely wants the
  iframe support in CCD too.
- The `claudemcpcontent.com` sandbox CSP limitation (#40) is a real constraint
  every MCP app has to work around. Data URIs are the reliable escape hatch
  for images; for scripts, inline them; for fonts, embed as data URIs too.
